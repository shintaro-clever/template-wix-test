# Role Boundary

## 目的
- 外部 AI、GitHub / repo、Wix Studio 担当者の役割境界を先に固定し、運用事故を防ぐ
- 「最初以外コードを触りたくない」前提と矛盾しない形で、どこまでをコード管理し、どこからを Wix Studio で仕上げるかを明確にする

## AI が作る範囲
- 静的原型の構造案
- HTML / CSS ベースの最小検証ページ
- 見出し、本文、CTA の文言素案
- 編集性チェックリスト、持ち込み runbook、役割境界などの運用文書

## repo で管理する範囲
- `prototype/` 配下の静的原型
- `docs/wix/` 配下の方針、仕様、手順、チェックリスト
- Wix 接続前提の最小補助 script と技術メモ
- 持ち込み前に参照する構造案と変更履歴

## Wix Studio で人が仕上げる範囲
- 実際のページ構成と最終レイアウト調整
- 画像、文言、CTA、余白、配置の最終判断
- セクション順の調整と公開前の見た目確認
- 実案件ページへの展開と運用判断

## 運用開始後にコードを触る必要がないケース
- テキストの修正
- 画像差し替え
- CTA 文言やリンク先の変更
- セクション順の変更
- 余白、配置、見た目の微調整

## 運用開始後にコードを触る可能性があるケース
- 新しい静的原型を追加で作るとき
- Wix Studio へ持ち込む前の構造案そのものを作り直すとき
- 運用ルール、検証仕様、手順書を更新するとき
- Wix Studio 側の編集では吸収しづらい大幅な構造変更を事前検証するとき

## 境界の考え方
- AI は「持ち込み元の原型」と「判断材料」を作る
- repo は「再利用できる原型と文書」を保持する
- 最終仕上げと日常運用は Wix Studio 担当者が行う
- 本番運用に入った後は、通常の更新を Wix Studio 側で完結させる前提を優先する
